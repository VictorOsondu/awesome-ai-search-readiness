# Contributing

Thanks for considering a contribution. This list is opinionated — that's the point. The curator has final say on inclusion, and "no" is not personal.

## What gets in

A resource is a good fit if **all** of these are true:

- A practitioner has used it and can vouch for it (not just "looks useful")
- It is current — published or meaningfully updated within the last 18 months, or genuinely evergreen
- It is concrete and actionable — teaches a method, ships a tool, presents evidence
- It moves the reader from *monitoring the problem* to *fixing the problem*
- It is free, or has a meaningful free tier (paid tools are welcome but must be labelled `[paid]`)

## What does not get in

- Vendor blog posts that are 80% product pitch
- Listicles that cite other listicles with no first-hand insight
- "Ultimate guide" content without an evidence base
- Resources older than 18 months unless explicitly evergreen
- Anything that reads as machine-generated and unedited
- Hype pieces without practical anchor — "AI will change everything" with no method attached

## Entry format

Each entry is a single line:

```
- [Title](https://url) — Description ending with a period. `[tag]` `[tag]`
```

Rules:
- Description ≤ 18 words
- End the description with a period
- Tags in backticks where useful: `[paid]`, `[requires signup]`, `[community]`, `[2026]`, `[2025]`, `[uk]`
- One blank line between subsections
- Alphabetical within a subsection unless a deliberate reading order is documented

## Pull request checklist

Before opening a PR:

1. The link resolves (no 404, no soft 404, no paywall surprise)
2. The entry fits one section — pick the most specific one
3. You wrote the description yourself, in your own words
4. You added a one-line justification in the PR body: *why this belongs and what it teaches*
5. You confirm you have no commercial relationship with the resource, or you have disclosed it

## House rules

- No self-promotion as the primary submission. If you maintain something useful, someone else will eventually submit it. If they don't, that's signal.
- One resource per PR. Bulk PRs get split or closed.
- Disagreements about scope are welcome in an issue, not in a forced PR.
